One-Page Guide

옵시디언 볼트 구조,
이 한 장만 보면 되는 버전

이 볼트는 자료를 쌓는 곳이 아니라 지금 할 일과 나중에 볼 것을 구분해서 움직이기 쉽게 만든 작업 공간입니다. 헷갈릴 때는 아래 세 가지만 기억하면 됩니다: 일정은 한 파일만 수정, 프로젝트는 상태별로 넣기, 오래된 건 아카이브로 보내기.

1. 폴더는 이렇게 나뉨

00_Inbox 일단 던지는 곳. 아직 정리 안 된 메모와 임시 노트.
10_Capture / 20_Learn / 30_Think 자료 저장, 공부 정리, 내 생각 정리를 분리하는 구역.
40_Projects 실제 프로젝트를 상태별로 관리하는 핵심 폴더.
50_Strategy / 60_Journal / 70_Sessions 계획, 작업일지, AI 세션 기록이 모이는 운영 구역.

2. 프로젝트 폴더는 이제 3갈래만 보면 됨

10_Active 지금 실제로 밀고 있는 프로젝트. 예: 베이비트래커
20_Backlog 보류했지만 버리지 않은 프로젝트. 예: Pomory, 버튜버외주, 인스타툰자동생성기
30_Writing 앱/툴보다는 글쓰기 중심 프로젝트. 예: 글쓰기어시스턴트
규칙: 새 프로젝트를 만들 때는 프로젝트 이름보다 먼저 상태를 정합니다.

3. 새 프로젝트 추가할 때 순서

1
일정 먼저 추가 50_Strategy/2026_ProjectTimeline.md에 프로젝트와 스케줄을 넣습니다. 이 파일이 원본입니다.
2
상태 폴더 선택 40_Projects/10_Active, 20_Backlog, 30_Writing 중 한 곳에 프로젝트 폴더를 만듭니다.
3
설명 노트만 붙이기 필요하면 README나 작업 노트를 넣습니다. 처음부터 문서를 많이 만들 필요는 없습니다.
4
시각화는 나중 2026_ProjectTimeline.html은 보기용 시각화라서, 원본 수정이 끝난 뒤 필요할 때만 맞추면 됩니다.

4. 헷갈리지 않으려면 이것만 기억

수정 원본은 1개
일정과 프로젝트 추가는 항상 2026_ProjectTimeline.md부터 합니다.
시각화는 보기 전용
2026_ProjectTimeline.html은 발표용, 확인용입니다. 기준 파일이 아닙니다.
죽은 문서는 삭제보다 아카이브
지금 안 쓰는 문서는 80_Archive나 _system/_archive로 보내서 동선에서 치웁니다.
시스템 문서는 _system
현재 쓰는 운영 문서만 두고, 예전 가이드는 아카이브로 보냅니다.

5. 지금 실제 기준 파일

00_Dashboard.md
전체 진입점
40_Projects/README.md
프로젝트 폴더 사용 규칙
_system/README.md
시스템 문서 안내
_system/SETUP_GUIDE.md
자동화와 작업일지 설명
50_Strategy/2026_ProjectTimeline.md
일정 원본

6. 한 줄 요약

뭔가 새로 시작할 때는 전략 파일에 일정 추가 → 프로젝트 폴더 생성 → 필요한 노트만 최소한으로 추가. 예전 문서가 보이면 삭제 고민보다 먼저 아카이브로 옮기면 됩니다.

기억할 핵심: 이 볼트는 “문서를 많이 만드는 시스템”이 아니라 “지금 하는 것과 나중에 볼 것을 쉽게 구분하는 시스템”입니다. 그래서 새 작업이 생기면 먼저 50_Strategy/2026_ProjectTimeline.md를 고치고, 프로젝트는 40_Projects 안에서 상태별로만 배치하면 거의 안 꼬입니다.
